# AI Agent 旅游行程规划助手 — 技术方案

## 1. 项目概述
工作目录：/src/


### 1.1 项目定位

基于 LLM 大模型的智能旅游行程规划助手，用户输入目的地、天数、偏好等需求，系统自动生成结构化的多日旅行计划（`ITravelPlan`），包含每日行程、景点推荐、住宿建议、天气信息、路线规划等。

### 1.2 技术选型结论

| 维度               | 选择                            | 理由                                                                 |
| ---------------- | ----------------------------- | ------------------------------------------------------------------ |
| **编排框架**         | **LangGraph**                 | 状态机模型天然适配多步工作流，支持条件路由、并行执行、Human-in-the-loop、Checkpoint 调试         |
| **Agent 架构**     | **多 Agent**                   | 不同职责用不同 Agent 承担，各自独立配置 LLM/Tools/Prompt，支持并行执行降低延迟和成本             |
| **运行时**          | Node.js + TypeScript          | 与项目现有 Monorepo (`@repo/agents`) 技术栈一致，复用 `@repo/shared-types` 类型定义 |
| **LLM Provider** | DeepSeek (DeepSeek-Reasoner / DeepSeek-V3) | DeepSeek-Reasoner 用于需要强推理的节点（规划/格式化），DeepSeek-V3 用于通用任务（意图理解/数据填充），整体成本远低于 GPT-4o 系列      |
| **外部服务**         | 高德地图 Web 服务 API               | POI 搜索、路径规划、酒店搜索；天气 API 补充气候数据                                     |

### 1.3 为什么不用其他方案

| 方案                         | 缺点                                        |
| -------------------------- | ----------------------------------------- |
| **LangChain 基础 Chain**     | 单次 LLM 调用无法插入 Tool Call，输出质量不可控，不适合复杂多步流程 |
| **Deep Agents 多 Agent 协作** | Agent 间通信协议复杂、调试困难、延迟高，对于本场景属于过度设计        |

## 3. 核心类型定义

### 3.1 数据模型

详见 `packages/shared-types/src/travel.ts` 和 `packages/agents/doc/数据结构/类型定义.md`。

核心接口：

```
ITravelPlan
├── planName: string
├── totalDays: number
├── totalDistance: number
├── vehicleType / vehicleAdvice / bestSeason / essentialItems
├── weather: IWeather[]
│   ├── area: string
│   ├── daytime: IWeatherDay { tempMax, tempMin, weather }
│   ├── nighttime: IWeatherDay { tempMax, tempMin, weather }
│   └── clothing: string
└── days: ITravel[]
    ├── day / title / waypoints / description
    ├── accommodation: IAccommodation[] (name, address, feature, booking?, price?)
    ├── foodRecommendation: string[]
    ├── commentTips?: string
    ├── activities: IActivity[]
    │   ├── name / description / suggestedHours
    │   ├── ticketPriceCny / openingHours
    │   └── images: { description, imgSrc }[]
    └── distance / drivingHours
```

JSON Schema 文件：`packages/agents/doc/数据结构/travel-plan.schema.json`

### 3.2 Graph State 定义

```typescript
import { Annotation, messagesStateUpdater } from "@langchain/langgraph"

export const TravelStateAnnotation = Annotation.Root({
  // ====== 输入 ======
  userInput: Annotation<string>({
    reducer: (current, _) => current,
    default: () => "",
  }),

  // ====== IntentAgent 输出 ======
  intent: Annotation<TravelIntent | null>({
    reducer: (_, update) => update,
    default: () => null,
  }),

  // ====== RoutePlanner 输出行程骨架 ======
  routeSkeleton: Annotation<RouteSkeletonDay[] | null>({
    reducer: (_, update) => update,
    default: () => null,
  }),

  // ====== POIAgent 输出 ======
  enrichedActivities: Annotation<Map<number, IActivity[]>>({
    reducer: (current, update) => {
      if (!update) return current ?? new Map()
      const merged = new Map(current)
      for (const [dayIdx, activities] of update) {
        merged.set(dayIdx, activities)
      }
      return merged
    },
    default: () => new Map(),
  }),

  // ====== WeatherAgent 输出 ======
  enrichedWeather: Annotation<IWeather[]>({
    reducer: (_, update) => update,
    default: () => [],
  }),

  // ====== HotelAgent 输出 ======
  enrichedAccommodation: Annotation<Map<number, IAccommodation[]>>({
    reducer: (current, update) => {
      if (!update) return current ?? new Map()
      const merged = new Map(current)
      for (const [dayIdx, hotels] of update) {
        merged.set(dayIdx, hotels)
      }
      return merged
    },
    default: () => new Map(),
  }),

  // ====== Formatter 输出 ======
  finalPlan: Annotation<ITravelPlan | null>({
    reducer: (_, update) => update,
    default: () => null,
  }),

  // ====== 控制字段 ======
  retryCount: Annotation<number>({
    reducer: (_, update) => update,
    default: () => 0,
  }),
  errors: Annotation<string[]>({
    reducer: (current, update) => [...(current ?? []), ...(update ?? [])],
    default: () => [],
  }),

  // ====== 中间消息（用于调试/Human-in-the-loop）=====
  messages: Annotation<BaseMessage[]>({
    reducer: messagesStateUpdater,
    default: () => [],
  }),
})

// ====== 子类型定义 ======

interface TravelIntent {
  destination: string
  days: number
  month: string
  travelType: string
  budget?: string
  travelers?: string
  preferences?: string[]
}

interface RouteSkeletonDay {
  day: number
  title: string
  waypoints: string
  description: string
  activities: Array<{
    name: string
    description: string
    suggestedHours: string
  }>
  accommodation: Array<{
    name: string
    address: string
    feature: string
  }>
  foodRecommendation: string[]
  commentTips?: string
}
```

### 3.3 State 设计要点

| 字段                      | 写入者          | 读取者                         | 说明                     |
| ----------------------- | ------------ | --------------------------- | ---------------------- |
| `userInput`             | 外部输入         | IntentAgent                 | 用户原始请求                 |
| `intent`                | IntentAgent  | RoutePlanner                | 结构化意图                  |
| `routeSkeleton`         | RoutePlanner | POI/Weather/Hotel/Formatter | 行程骨架                   |
| `enrichedActivities`    | POIAgent     | Formatter                   | 按 day index 存储已丰富的活动数据 |
| `enrichedWeather`       | WeatherAgent | Formatter                   | 完整天气数组                 |
| `enrichedAccommodation` | HotelAgent   | Formatter                   | 按 day index 存储住宿数据     |
| `finalPlan`             | Formatter    | Validator / API Response    | 最终输出                   |
| `retryCount`            | Validator    | 条件路由判断                      | 控制重试次数                 |

## 4. Agent 详细设计

### 4.1 IntentAgent — 意图理解

**角色**: 将用户的自然语言输入转化为结构化的旅行需求。

**LLM 配置**: `deepseek-v3` (temperature: 0.3)

**Tools**: 无（纯 LLM 推理）

**System Prompt 要点**:

- 从用户输入中提取：目的地、天数、时间、出行方式、预算、同行人数、特殊偏好
- 对于模糊信息给出合理默认值（如未指定天数则根据目的地推荐）
- 如果关键信息缺失，标记为 `needsClarification`

**输入/输出**:

```
输入: State.userInput
输出: State.intent (TravelIntent)
```

**Prompt 示例**:

```
你是一个专业的旅行顾问。请从用户的需求描述中提取结构化信息。

必须提取的字段：
- destination: 目的地（如"新疆"、"云南"）
- days: 计划天数（数字，如果用户没说就根据目的地推荐合理天数）
- month: 出行月份或季节
- travelType: 出行方式（"自驾"、"跟团"、"自由行"等）
- budget: 预算范围（可选）
- travelers: 同行人员（可选）
- preferences: 特殊偏好数组（可选，如["摄影"、"美食","避人流"]）

规则：
1. 不要编造用户没有提到的信息，缺失字段留空
2. days 必须是具体数字
3. 如果信息严重不足导致无法规划，在 response 中说明需要补充什么
```

***

### 4.2 RoutePlanner Agent — 行程骨架生成

**角色**: 基于用户意图生成完整的行程骨架（含景点名称、大致描述），但不包含具体的门票价格、开放时间等需 API 查询的数据。

**LLM 配置**: `deepseek-reasoner` (temperature: 0.7 — 需要一定创造性来设计有趣路线)

**Tools**: 无（纯 LLM 推理，利用自身知识库）

**System Prompt 要点**:

- 根据 intent 生成合理的行程路线
- 每天的景点安排考虑地理位置相邻性（减少路程时间）
- 合理分配每天的游玩强度（避免某天过于紧凑）
- 包含当地特色美食推荐
- 给出实用的 commentTips（穿衣、注意事项）

**输入/输出**:

```
输入: State.intent
输出: State.routeSkeleton (RouteSkeletonDay[])
```

**Output Format 要求**:

```typescript
// 必须严格按照此格式输出 JSON
type RouteSkeletonDay = {
  day: number
  title: string
  waypoints: string       // JSON字符串: ['地点A','地点B']
  description: string
  activities: Array<{
    name: string           // 景点名称（后续用于高德POI精确匹配）
    description: string
    suggestedHours: string
  }>
  accommodation: Array<{
    name: string
    address: string
    feature: string
  }>
  foodRecommendation: string[]
  commentTips?: string
}
```

### 4.3 POIAgent — 景点信息丰富

**角色**: 为行程骨架中的每个景点补充详细信息：通过高德 API 查询精确位置、门票价格、开放时间、参考图片等。

**LLM 配置**: `deepseek-v3` (temperature: 0.3 — 机械性工作，不需要创造力)

**Tools**:

| Tool 名称             | 功能                    | API 来源            |
| ------------------- | --------------------- | ----------------- |
| `search_poi`        | 关键词搜索 POI，获取经纬度、地址、分类 | 高德搜索 POI          |
| `get_poi_detail`    | 获取 POI 详情（电话、评分等）     | 高德详情查询            |
| `get_route_driving` | 两点间驾车距离和时间            | 高德驾车规划            |
| `get_ticket_price`  | 获取景点门票价格              | 内部知识库 / 高德扩展信息    |
| `search_images`     | 搜索景点参考图片              | 可选（Unsplash/自建图床） |

**System Prompt 要点**:

- 遍历 routeSkeleton 中每一天的每个 activity
- 对每个景点调用 search\_poi 获取精确信息
- 用 get\_route\_driving 计算当天总距离和驾驶时间
- 如果某个景点在高德查不到，保留原始 name/description 并标记 ticketPriceCny 为 0

**输入/输出**:

```
输入: State.routeSkeleton
输出: State.enrichedActivities (Map<dayIndex, IActivity[]>)
       State.distance / State.drivingHours (回写到每条 ITravel)
```

**并行策略**:

- 每天的景点可以并行查询
- 不同天之间完全独立，可并行

***

### 4.4 WeatherAgent — 天气信息

**角色**: 根据目的地和出行季节，查询并生成各区域的天气预报和穿衣建议。

**LLM 配置**: `deepseek-v3` (temperature: 0.1 — 几乎是确定性任务)

**Tools**:

| Tool 名称                  | 功能                 |
| ------------------------ | ------------------ |
| `get_weather_forecast`   | 查询指定城市/区域的天气预报     |
| `get_historical_weather` | 查询历史同期天气（用于未来日期预测） |

**System Prompt 要点**:

- 从 routeSkeleton 的 waypoints 中提取涉及的区域/城市
- 为每个区域查询白天/夜间天气
- 根据温度和天气状况给出穿衣建议
- 注意不同区域的温差（如山区 vs 盆地）

**输入/输出**:

```
输入: State.routeSkeleton (提取区域列表)
输出: State.enrichedWeather (IWeather[])
```

***

### 4.5 HotelAgent — 住宿推荐

**角色**: 为每天的行程终点推荐合适的住宿。

**LLM 配置**: `deepseek-v3` (temperature: 0.3)

**Tools**:

| Tool 名称                 | 功能             |
| ----------------------- | -------------- |
| `search_hotel`          | 在指定位置附近搜索酒店/民宿 |
| `get_hotel_price_range` | 获取该地区住宿的大致价格区间 |

**System Prompt 要点**:

- 根据每天的目的地搜索附近住宿
- 推荐 1-3 家不同档次的选项
- 包含预订建议（是否需要提前预订）
- 价格从高德 API 或内部数据库获取

**输入/输出**:

```
输入: State.routeSkeleton (每天的目的地)
输出: State.enrichedAccommodation (Map<dayIndex, IAccommodation[]>)
```

***

### 4.6 Formatter Agent — 最终组装

**角色**: 将所有 Agent 收集的数据组装成严格符合 `ITravelPlan` 接口的最终 JSON。

**LLM 配置**: `deepseek-reasoner` (temperature: 0 — 需要 100% 符合 Schema)

**Tools**: 无

**System Prompt 要点**:

- 这是整个流程的最后一步，输出必须严格符合 ITravelPlan JSON Schema
- 将 routeSkeleton 作为基础框架
- 用 enrichedActivities 替换 skeleton 中的 activities（使用 API 返回的完整数据）
- 用 enrichedWeather 直接填入 weather 字段
- 用 enrichedAccommodation 替换 skeleton 中的 accommodation
- 确保 totalDistance 是所有 days distance 的累加
- 计算 vehicleAdvice 和 essentialItems

**输入/输出**:

```
输入: State.routeSkeleton + State.enrichedActivities
     + State.enrichedWeather + State.enrichedAccommodation
输出: State.finalPlan (ITravelPlan)
```

**强制约束手段**:

- 使用 `StructuredOutputParser` 或 `JsonOutputKeyToolsParser` 强制 LLM 输出合法 JSON
- 后接 Zod Schema 二次校验作为双保险

const AMAP_KEY = process.env.AMAP_WEB_SERVICE_KEY!

    type: poi.type,
  })) ?? []
}, {
  name: "search_poi",
  description: "根据关键词在城市内搜索POI（景点、餐厅、酒店等）。参数: keywords(必填), city(可选)",
  schema: z.object({
    keywords: z.string().describe("搜索关键词，如'喀纳斯景区'"),

### 5.3 错误处理与降级

```typescript
// 每个 Tool 统一的错误包装
async function safeToolCall<T>(fn: () => Promise<T>, fallback: T): Promise<T> {
  try {
    return await fn()
  } catch (error) {
    logger.warn(`Tool call failed, using fallback`, { error })
    return fallback
  }
}
```

降级策略：

- **高德 API 限流**: 返回空数组，Formatter Agent 会 fallback 到 RoutePLanner 的原始数据
- **网络超时**: 3s 超时后返回 fallback
- **API Key 失效**: 告警 + 返回空数据

## 6. Graph 定义

### 6.1 完整图定义

```typescript
// packages/agents/src/graph/index.ts
import { StateGraph } from "@langchain/langgraph"
import { TravelStateAnnotation } from "./state.js"

const graph = new StateGraph(TravelStateAnnotation)

// ========== 节点注册 ==========
graph.addNode("intent_agent", intentAgentNode)
graph.addNode("route_planner", routePlannerNode)
graph.addNode("poi_agent", poiAgentNode)
graph.addNode("weather_agent", weatherAgentNode)
graph.addNode("hotel_agent", hotelAgentNode)
graph.addNode("formatter", formatterNode)
graph.addNode("validator", validatorNode)

// ========== 边定义 ==========
graph.addEdge(START, "intent_agent")
graph.addEdge("intent_agent", "route_planner")

// Fan-out: RoutePlanner 完成后并行分发到三个 Agent
graph.addEdge("route_planner", "poi_agent")
graph.addEdge("route_planner", "weather_agent")
graph.addEdge("route_planner", "hotel_agent")

// Fan-in: 三个 Agent 都完成后进入 Formatter
graph.addEdge("poi_agent", "formatter")
graph.addEdge("weather_agent", "formatter")
graph.addEdge("hotel_agent", "formatter")

graph.addEdge("formatter", "validator")

// 条件边: 校验结果决定下一步
graph.addConditionalEdges(
  "validator",
  shouldRetryOrEnd,
  {
    retry: "route_planner",  // 校验失败 → 回退重试
    success: END,            // 校验通过 → 结束
  }
)

// ========== 编译 ==========
export const travelPlannerGraph = graph.compile({
  interruptBefore: [],   // Human-in-the-loop: 可在此处打断
  interruptAfter: ["route_planner", "formatter"], // 允许用户在这些节点后介入修改
  checkpointer: memorySaver, // Checkpoint 支持
})
```

### 6.2 条件路由函数

```typescript
function shouldRetryOrEnd(state): "retry" | "success" {
  if (!state.finalPlan) return "retry"
  if (state.retryCount >= 2) {
    // 超过最大重试次数，标记警告但仍然返回（降级模式）
    state.errors.push("达到最大重试次数，当前结果可能不完整")
    return "success"
  }

  // Zod 校验
  const result = TravelPlanSchema.safeParse(state.finalPlan)
  if (result.success) return "success"

  // 校验失败 → 重试
  state.errors.push(result.error.message)
  return "retry"
}
```

## 7. 流式输出与前端对接

### 7.1 SSE 进度推送

```typescript
// apps/apis/src/router/agentRouter.ts
router.post("/plan", async (ctx) => {
  const { input } = ctx.request.body as { input: string }

  const config = { configurable: { thread_id: uuidv4() } }
  const stream = await travelPlannerGraph.stream(
    { userInput: input },
    { ...config, streamMode: "updates" },
  )

  ctx.set("Content-Type", "text/event-stream")
  ctx.set("Cache-Control", "no-cache")

  for await (const event of stream) {
    const nodeName = Object.keys(event)[0]
    const nodeData = event[nodeName]

    // 推送进度事件
    ctx.res.write(`event: progress\n`)
    ctx.res.write(`data: ${JSON.stringify({ node: nodeName, data: nodeData })}\n\n`)
  }

  // 最终结果
  const finalState = await travelPlannerGraph.getState(config)
  ctx.res.write(`event: complete\n`)
  ctx.res.write(`data: ${JSON.stringify(finalState.values.finalPlan)}\n\n`)
  ctx.body = ""
})
```

### 7.2 前端事件类型

| Event Type        | 触发时机       | Data 内容                                   |
| ----------------- | ---------- | ----------------------------------------- |
| `progress`        | 每个 Node 完成 | `{ node: "intent_agent", data: {...} }`   |
| `human-interrupt` | 打断点等待用户确认  | `{ node: "route_planner", state: {...} }` |
| `complete`        | 全部完成       | `ITravelPlan` JSON                        |
| `error`           | 发生错误       | `{ message: "...", code: "..." }`         |

## 9. 错误处理与重试机制

### 9.1 重试策略

```
RoutePlanner → Formatter → Validator
                               │
                    ┌──────────┴──────────┐
                    ▼                      ▼
                 通过 (success)          失败 (retry)
                    │                      │
                    ▼                      ├─ retryCount < 2 → 回到 RoutePlanner
                   END                     │
                                            └─ retryCount >= 2 → 降级返回 + 警告信息
```

### 9.2 各层级错误处理

| 层级              | 错误类型                 | 处理方式                   |
| --------------- | -------------------- | ---------------------- |
| **LLM 调用失败**    | Rate limit / Timeout | 自动重试 3 次，指数退避          |
| **Tool 调用失败**   | 高德 API 错误            | 返回 fallback 空值，不影响整体流程 |
| **Schema 校验失败** | 格式不符合 ITravelPlan    | 重试整个规划流程（最多 2 次）       |
| **全流程超时**       | 总耗时 > 120s           | 中止，返回已有部分结果 + 超时提示     |
